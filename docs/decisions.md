# Decisions Log

## 2026-07-07 — Stack & Architecture

- Python 3.11+ chosen over Node.js — matches existing pattern for API/data tooling (vs. Node for Astro/frontend work)
- Typer + Rich for CLI — clean commands now, room to grow subcommands later without becoming unreadable
- Provider abstraction (BaseImageProvider) built from day one, even with only one model — keeps future model additions to "one new file" instead of a rewrite
- Reference images organized per-campaign subfolder under reference-images/
- Output kept flat in output/ — organization handled by manifest.json + filenames, not folder nesting
- CLI structured as subcommands (generate, list-refs, history) over a single flag-heavy command — more discoverable, room to add commands later
- Default 3 image variations per generation call, not 1 — optimizing for "pick the best" workflow
- Filename convention: {campaign}_{timestamp}_{n}.png

## 2026-07-08 — Prompting, Reliability & CLI Behavior

- **Brand guide injection**: `reference-images/brand-guide.md` content is automatically prepended to every prompt sent to the model — never manually pasted per-call. The manifest logs the full assembled prompt (brand guide + user prompt), not just the raw `--prompt` text, so history reflects what was actually sent to the API.
- **Reference image cap (14 max)**: if a campaign folder has more than 14 images and `--refs` is omitted, auto-truncate to the first 14 in alphabetical filename order, print a console warning, and log only the images actually used in the manifest's `reference_images` field.
- **Partial success**: `GenerationResult` carries per-image outcomes (not a single batch-level bool). Manifest `status` is one of `success` | `partial` | `failed`. On partial failure, successfully generated images are kept and saved; failed images are logged with their error, and the call is not treated as a total failure.
- **Retry/backoff**: exponential backoff (2s, 4s, 8s), max 3 retries per image, applied to HTTP 429 and 5xx/timeout errors. Non-retryable errors (e.g. 400) fail immediately without consuming a retry.
- **Reference image formats**: `.png`, `.jpg`/`.jpeg`, `.webp` accepted. Any other file in a campaign folder (notes, `.gif`, etc.) is silently skipped during auto-collection — not an error.
- **Cost guardrail**: `--count` hard-capped at 10; values above that are rejected with a clear error. No spend/cost tracking in the manifest — cap only.
- **Testing**: light `pytest` coverage for deterministic, non-visual logic only — manifest I/O, filename collision-suffix naming, param validation (aspect ratio/resolution allowlists), backoff timing. No tests for actual image generation output (inherently manual/visual).
- **Console UX**: Rich progress bar/spinner per image during generation; visible retry messages (e.g. "Rate limited, retrying in 4s... (attempt 2/3)"); a per-image summary table (✓/✗ + reason) at the end of each call; warnings/errors color-coded via Rich.
- **Multi-turn conversational editing**: deferred to phase 2. Not built in v1 (`generate` calls remain independent/stateless), but manifest and provider design should stay mindful of eventually needing a session/parent-generation concept.
- **Campaign folder creation**: manual (`mkdir`) only — no CLI scaffolding command. A `generate` call against a campaign folder that doesn't exist at all is a hard error (distinct from an existing-but-empty folder, which remains valid for text-only generation per the original spec).
- **Logging**: `output/manifest.json` is the sole persistent record. No separate debug/log file — this is an interactive local tool, so crashes/tracebacks print directly to the terminal.
- **`history` command**: sorted most-recent-first. Prompt preview column shows only the user-typed `--prompt` text truncated to ~60 chars (not the brand-guide-injected full prompt, which stays in the manifest JSON for full detail). Output files column shows filenames only, not full paths.
- **Git tracking**: `.gitignore` excludes `.env`, `venv/`, `output/*.png`, `__pycache__/`, `*.pyc`. Tracks `output/manifest.json`, `reference-images/**`, and all code/config/docs. (Note: `.gitignore` did not actually exist prior to this session despite being referenced as done in progress notes — needs to be created.)
- **No auto-open of generated images** after a `generate` call — file paths are printed in the summary table; opening `output/` is on the user. May add an opt-in `--open` flag later.
- **`list-refs` output**: no args → table of campaign folders with image counts, alphabetical. `--campaign X` → filenames within that campaign (image formats only, non-images silently omitted), with a note if the folder exceeds 14 images (ties to the truncation behavior above).
- **Manifest path format**: all paths stored as forward-slash strings regardless of host OS, matching the schema example.
- **API key validation**: missing/invalid `GEMINI_API_KEY` fails fast with a clear error message before any API call is attempted.

## 2026-07-08 — Phase 1 Implementation Notes

- **Manifest carries both `prompt` and `user_prompt`**: `prompt` is the full brand-guide-injected text actually sent to the model (per the Q1 brand-injection decision); `user_prompt` is just the raw `--prompt` text. This resolves a tension between Q1 (log the full assembled prompt) and Q12 (`history`'s preview column should show only the user's own text, not the brand-guide boilerplate repeated on every row) — without `user_prompt`, `history` would have had to fragilely guess where the brand guide ends and the user's text begins.
- **SDK call shape** (confirmed by introspecting the installed `google-genai==2.10.0`, not guessed): `client.models.generate_content(model=..., contents=[prompt, *PIL.Image objects], config=types.GenerateContentConfig(response_modalities=["TEXT","IMAGE"], image_config=types.ImageConfig(aspect_ratio=..., image_size=...)))`. Output image extracted via `part.as_image()` on `response.parts`. The SDK also exposes a newer `client.interactions.create()` surface — deliberately not used, as it's a much larger agentic API (steps, tools, MCP) that's unnecessary complexity for straightforward image generation.
- **`config/settings.yaml` gained a `model_string` key** (e.g. `gemini-3-pro-image`) — CLAUDE.md's original config schema didn't spell this out explicitly but did require "the exact model string isolated in config, never hardcoded," which needed a concrete key to live in.
- **Retry/backoff and per-image progress messaging live in `providers/nano_banana_pro.py` itself** (using `rich.console.Console` directly), not routed through a callback into `main.py`. Kept the `BaseImageProvider.generate()` abstract signature unchanged from its original design (prompt, campaign, reference_images, count, aspect_ratio, resolution, output_dir) rather than threading a progress-callback parameter through it.
- **CLI table titles print as a plain `console.print()` line above the table, not via `Table(title=...)`**: Rich sizes a title's wrapping to the table's own rendered width, so a long title on a narrow (few-column or empty) table wraps character-by-character. Also switched `history`'s `Timestamp`/`Outputs` columns to `overflow="fold"` instead of the default ellipsis truncation — on this Windows setup, Python emitted the ellipsis as a single-byte CP1252 character that the terminal displayed as a replacement glyph. Folding avoids depending on that Unicode character rendering correctly at all.

## 2026-07-10 — Prompt Refinement Step (`--refine/--no-refine`)

- **Feature**: before a `--prompt` reaches the image model, `generate` now runs an interactive refinement step by default — identifies ambiguities in the raw prompt and asks clarifying questions (each with a recommended on-brand default; Enter accepts it), then synthesizes the prompt + answers into one polished paragraph, shown for confirmation before being brand-guide-assembled and sent to the image model. `--no-refine` skips it entirely. Motivated by Rahul's observation that prompts were going straight from him to the image model with no improvement/polishing step at the project level.
- **On by default, not opt-in**: matches the stated goal of catching every prompt, not just ones Rahul remembers to flag.
- **Text-only refiner, not multimodal**: the refiner reasons from the raw prompt, campaign name, reference image *filenames* (not the actual images), and brand-guide.md content. Filenames alone are informative (e.g. `Bhujangasana (Cobra Pose).png`). Chosen over multimodal for speed/cost — a clarifying-questions step shouldn't itself be a slow, expensive call.
- **Separate cheap text model** (`refiner_model_string: gemini-2.5-flash` in settings.yaml), never the expensive image model (`model_string: gemini-3-pro-image`), isolated in config the same way per the existing model-string convention.
- **Two-call design, second call skipped when unnecessary**: clarifying-questions call (structured JSON output via `response_schema`/`response_mime_type="application/json"`, returns `[]` when the prompt is already unambiguous) → synthesis call (plain paragraph, only made if there were questions to synthesize from). Capped at `max_refinement_questions: 4` in settings.yaml.
- **Confirmation gate**: the synthesized paragraph is shown to Rahul before it's sent to the image model; declining aborts the `generate` call cleanly — no image API call made, no manifest entry written (raised before `output_dir.mkdir()`/provider construction).
- **Fail-open policy**: `providers/prompt_refiner.py`'s two methods raise on unrecoverable API failure (matching how `providers/nano_banana_pro.py` already behaves); `main.py`'s new `_refine_prompt` orchestration function catches those exceptions, prints a Rich warning, and falls back to the raw `--prompt` text — refinement can never block or crash a `generate` call. If synthesis fails after questions were already answered, `refinement_qa` still records what was asked/answered even though `refined_prompt` is null, since that's strictly more informative than discarding it.
- **Retry logic extracted to `utils/retry.py`** (`RETRY_DELAYS`, `is_retryable`, `call_with_retry`), reused by both `providers/nano_banana_pro.py` and the new `providers/prompt_refiner.py`. Previously this lived inline only in `nano_banana_pro.py` per the "no premature abstraction" philosophy (only 1 call site existed); now that there are genuinely 2+ call sites needing identical behavior, extraction is justified. `nano_banana_pro.py::_call_with_retry` is now a thin delegating wrapper; behavior unchanged, covered by `tests/test_nano_banana_pro.py`'s delegation test plus the retargeted `tests/test_retry.py`.
- **Manifest schema gains `refined_prompt` and `refinement_qa`** — see CLAUDE.md's Manifest Schema section for the exact semantics (both are null/`[]` when refinement was skipped, unambiguous, or failed). `user_prompt` and `prompt` keep their pre-existing meaning; `prompt` is now built from `refined_prompt` when refinement produced one.
- **`pydantic` added as an explicit `requirements.txt` dependency** — previously only an implicit transitive dependency of `google-genai`; now imported directly in `providers/prompt_refiner.py` for `BaseModel`/`response_schema`.
- **New tests**: `tests/test_prompt_refiner.py` (mocks `client.models.generate_content`, covering parsed-list return, empty-when-unambiguous, truncation to `max_questions`, the manual-JSON-parse fallback when `response.parsed` is `None`, retry-exhaustion raising, synthesis text stripping, non-retryable-error raising) and `tests/test_nano_banana_pro.py` (confirms the post-refactor retry delegation still works). `tests/test_retry.py` retargeted to test `utils/retry.py` directly instead of going through the provider. No new tests for `main.py`/`_refine_prompt` itself — matches existing precedent of zero direct `main.py` tests, since the interactive Rich-prompt flow is explicitly out of this project's testing scope (deterministic logic only).
- **Process note**: this feature was planned via a Plan subagent that reported making live calls to the real Gemini API (using the project's `GEMINI_API_KEY`) to verify the `response_schema`/`response_mime_type` call shape and confirm `gemini-2.5-flash` exists and supports `generateContent` — an external, cost-incurring action taken without asking Rahul first. No files were changed by the agent (confirmed via `git status`), and `pydantic` being genuinely installed is consistent with its claims, but this wasn't verified with full certainty. Flagged transparently to Rahul; live-API verification will not be delegated to subagents without asking first going forward. The technical claims (SDK call shape, model existence) were treated as plausible-but-unverified and cross-checked against the installed SDK's type signatures before being relied on for implementation.

## 2026-07-12 — Decoupled Refiner Q&A: `refine-questions` + `--answers-file`

- **Problem being fixed**: the 2026-07-11 "Process note on refiner Q&A relay reliability" entry above documented that a Claude-driven relay of refiner questions to Rahul in chat is unreliable — probing `generate_questions()` once and piping answers into a later live `generate` call can land answers on the wrong questions, because the live call's own internal question-generation is non-deterministic and can return a different set. Rahul confirmed the fix direction: split "ask" and "answer" into two separate, non-interactive CLI steps.
- **`refine-questions` command added**: runs only the clarifying-questions half of refinement (no image generation, no manifest entry), printing the results and optionally writing them to `--out` as a JSON array of `{"question", "recommended_answer"}` objects.
- **`generate` gained `--answers-file`**: consumes that same JSON shape (with an `"answer"` filled in per item) and, when supplied, skips the live `generate_questions()` call and the interactive `Prompt.ask`/`Confirm.ask` loop entirely — going straight to synthesis. This is the actual fix: the live `generate` call can no longer drift onto a different question set than what was relayed, because it never generates questions in this path at all. Missing/blank `"answer"` values fall back to `"recommended_answer"`, mirroring the interactive Enter-to-accept-default behavior. Hard error if combined with `--no-refine`.
- **No manifest schema change** — `refined_prompt`/`refinement_qa` are populated identically regardless of whether the Q&A came from the interactive loop or the answers file.
- **`_resolve_campaign_refs` extracted** from `generate()`'s previously-inline campaign-dir-check + refs-resolution logic, now shared with `refine-questions`. Same precedent as the `utils/retry.py` extraction (2026-07-10 entry): duplicating identical logic across 2+ call sites isn't justified once a second genuine call site exists.
- **Answers-file length is not capped** against `max_refinement_questions` — that cap only bounds cost/UX on the *live* question-generation call, which this path skips; capping supplied answers here would silently drop them, recreating the exact bug this feature fixes.
- **Consistency between the answers file and the later `generate` call's `--prompt`/`--campaign` is not enforced** — doing so would require expanding the file schema from a bare array to an object carrying that metadata, which was judged out of scope for this fix. Documented as an operator-responsibility limitation in CLAUDE.md instead.
- **First direct test coverage of `main.py` logic** (`tests/test_main.py`): the new `_load_answers_file` and `_resolve_campaign_refs` helpers are pure/non-interactive, so they're unit tested directly, along with the non-interactive branch of `_refine_prompt` (mocking `PromptRefiner`, asserting `Prompt.ask`/`Confirm.ask` are never called). The Typer command bodies themselves (`generate`, `refine-questions`) remain untested, consistent with prior precedent — verified manually instead against the real API (see below).
- **Manually verified end-to-end against the real API**: `refine-questions` against `individual-yoga-poses` produced 4 real questions and wrote them to a JSON file; answers were edited in (including one deliberately blank and one key omitted entirely, both correctly falling back to `recommended_answer`); `generate --answers-file ... --count 1` ran with zero interactive prompts, synthesized correctly from the supplied answers, and the manifest's `refinement_qa` matched the file exactly. `--answers-file` combined with `--no-refine` errored immediately with no API calls made.
- **Second, more meaningful verification run — the actual intended workflow**: ran `refine-questions` against `yoga-excercises-infographics` for a real "12 Sun Salutation Steps" vector infographic prompt (9:16 portrait, orange accent, English+Sanskrit captions, title text). The 3 real clarifying questions were relayed to Rahul in chat, his answers (all recommended defaults, accepted as-is) were written into the answers file, and `generate --answers-file` produced a correct, on-brand result on the first attempt — no interactive prompts, no drift. Confirms the feature works in the actual Claude-driven-relay scenario it was built for, not just a scripted simulation.
